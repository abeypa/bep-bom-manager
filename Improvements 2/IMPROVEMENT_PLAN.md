## 1. Critical: Price History System

### Current Problem

Parts have a single `base_price` field. When someone updates a part's price, the old price is gone forever. There is no record of what a part cost last month, last quarter, or last year.

### Root Cause

No `part_price_history` table exists. The `update` operation on any part table simply overwrites `base_price`.

### Solution

**New table: `part_price_history`**

```sql
CREATE TABLE IF NOT EXISTS part_price_history (
  id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  part_table_name TEXT NOT NULL,
  part_id BIGINT NOT NULL,
  part_number TEXT NOT NULL,
  old_price NUMERIC(18, 2),
  new_price NUMERIC(18, 2) NOT NULL,
  old_currency TEXT,
  new_currency TEXT NOT NULL DEFAULT 'INR',
  old_discount_percent NUMERIC(5, 2),
  new_discount_percent NUMERIC(5, 2),
  change_reason TEXT,
  changed_by TEXT,
  changed_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

**How it works:**

Every time `base_price`, `currency`, or `discount_percent` changes on ANY part table, a row is inserted into `part_price_history`. The frontend intercepts every update, compares old values against new values, and writes the history row before applying the change. This captures changes from manual edits, JSON imports, and the "update master" checkbox in project BOM editing.

**Frontend changes needed:**

The `partsApi.updatePart()` function must fetch the current part first, compare price fields, and insert a history row if any changed. Similarly `partsApi.importParts()` must do the same comparison during bulk import. A new "Price History" button on each part row opens a modal showing the price timeline.

**What gets captured:**

Manual price edits, price changes via JSON import, price changes via the "update master" checkbox, discount percentage changes, currency changes, who made the change (user email from auth), and when (timestamp).

## 2. Critical: Project BOM Price Snapshot Fix

### Current Problem

When a part is added to a project BOM, the `unit_price` is snapshot correctly. But `discount_percent` is NOT stored in `project_parts`. Instead, it's read from the master part table at display time and PO generation time.

**Real-world consequence:** Supplier changes their discount from 10% to 5% after a project BOM is finalized. The PO generated from that old project will silently use the NEW 5% discount instead of the original 10%. Project cost calculations drift without anyone noticing.

### Where the Bug Lives

In `CreatePOFromBOMModal.tsx`, the discount comes from the current master part:
```typescript
const discount = p.catalogItem?.discount_percent || 0  // reads CURRENT master value
```

The `unit_price` comes from the snapshot, which is correct. But the total calculation uses `price * quantity * (1 - discount/100)` where `discount` is stale.

### Solution

**Step 1:** Add missing columns to `project_parts` table:

```sql
ALTER TABLE project_parts ADD COLUMN IF NOT EXISTS discount_percent NUMERIC(5,2) NOT NULL DEFAULT 0;
ALTER TABLE project_parts ADD COLUMN IF NOT EXISTS base_price_at_assignment NUMERIC(18,2);
ALTER TABLE project_parts ADD COLUMN IF NOT EXISTS supplier_name_at_assignment TEXT;
ALTER TABLE project_parts ADD COLUMN IF NOT EXISTS assigned_at TIMESTAMPTZ DEFAULT NOW();
```

**Step 2:** When `addPartToSection` runs, snapshot ALL price-relevant data: `unit_price` (already done), `discount_percent` (new), `currency` (already done), `base_price_at_assignment` (new), and `supplier_name_at_assignment` (new).

**Step 3:** In `CreatePOFromBOMModal.tsx`, change discount source from `p.catalogItem?.discount_percent` to `p.discount_percent` (from the project_parts snapshot).

**Step 4:** The "update master" checkbox should update the master part but never retroactively change existing project_parts rows.

## 3. Critical: Schema vs Code Mismatch

### Current Problem

The SQL scripts define `project_parts` with a polymorphic pattern using `part_type TEXT` and `part_id BIGINT`. But the frontend code uses individual FK columns: `mechanical_manufacture_id`, `mechanical_bought_out_part_id`, `electrical_manufacture_id`, `electrical_bought_out_part_id`, `pneumatic_bought_out_part_id`.

The TypeScript types in `database.ts` match the SQL (polymorphic), not the frontend. This means the SQL scripts in the repo will not recreate a working schema, TypeScript provides zero type safety for this table, and the `validate_project_part` trigger references the polymorphic pattern and will not work with individual FK columns.

### Solution

Since there's no data to preserve, rebuild the table using the individual FK pattern that the frontend already expects. This is the better approach because Supabase's `.select()` with embedded joins only works with real FK columns, it's more explicit, and it allows a CHECK constraint enforcing exactly one FK is set per row.

The rebuilt table includes individual FK columns with proper foreign key references, the price snapshot columns from fix #2, a `use_date_time` column the code already references, and a CHECK constraint ensuring exactly one part FK is populated per row.

The TypeScript types must be updated to match this schema, removing `part_type` and `part_id` and adding the individual FK columns plus new snapshot columns.

## 4. High: Stock Logic Fixes

### 4a. Negative Stock Prevention

When adding a part to a project, `stock_quantity` is decremented with no guard. If stock is 5 and someone assigns 10, stock becomes -5.

**Fix:** Add client-side validation before the database call:
```typescript
if (quantity > (part.stock_quantity || 0)) {
  throw new Error(`Insufficient stock. Available: ${part.stock_quantity}, Requested: ${quantity}`)
}
```

Also add database CHECK constraints as a safety net:
```sql
ALTER TABLE mechanical_manufacture ADD CONSTRAINT chk_mm_stock CHECK (stock_quantity >= 0);
```
Repeat for all 5 part tables.

### 4b. total_stock and total_amount Not Computed

Part tables have `total_stock` and `total_amount` columns but they are never automatically computed. They sit at 0 forever.

**Options:** Either add database triggers to compute them on INSERT/UPDATE, or remove them entirely and compute on-the-fly in the frontend when needed.

### 4c. Stock Restoration Audit Trail

When `removePartFromSection` runs, it restores the full quantity to master stock but doesn't log this event distinctly. Add a stock movement log entry with "STOCK-RESTORE" type so the audit trail shows why stock suddenly increased.

The dashboard calls `supabase.rpc('get_dashboard_stats')` but this function does not exist in the SQL scripts. The dashboard crashes on fresh deployment.

**Fix:** Create the RPC function (SQL provided in section 12) and add a fallback in `dashboardApi.getStats()` that uses direct count queries if the RPC fails. The fallback was implemented in the previous update.

## 6. Medium: PO Workflow Completion

### Problems Found

No PO detail view (clicking a PO row does nothing). No status change workflow. No delete capability. No item receiving flow. PO totals use current master discount instead of snapshot.

### Solution

Add a PO Detail Modal that shows full PO information including line items, supports editing notes and terms (when PO is Pending or Sent), shows valid status transitions with action buttons, has a "Receive Items" flow where users enter received quantities per line item (stock auto-updates), allows deleting the PO (only when Pending or Cancelled), and allows deleting individual line items.

Status transition rules should be enforced: Pending can go to Sent or Cancelled. Sent can go to Confirmed or Cancelled. Confirmed can go to Partial, Received, or Cancelled. Partial can go to Received or Cancelled. Received and Cancelled are terminal states.

PO number collision: add random suffix to timestamp-based numbers.

## 7. Medium: Part In / Part Out Feature

### Current State

Part usage logs exist but mix incoming and outgoing stock in one flat table with no clear distinction.

### Recommended Solution

Create a dedicated `stock_movements` table instead of overloading `part_usage_logs`. This table has a clear `movement_type` enum (IN, OUT, ADJUST), proper foreign key references to supplier and project tables, captures the price at time of movement, and can track manual stock adjustments.

**Part In flow:** User selects a part, selects supplier, enters received quantity and optional PO reference. Master part's stock_quantity increases. Movement is logged.

**Part Out flow:** Two views available. Grouped by project (expandable cards showing all consumed parts with totals per project). Flat chronological view of all outgoing stock movements.

## 9. Low: Code Quality and UX

**TypeScript type safety:** 40+ instances of `as any` casts throughout the API layer. After aligning the schema, regenerate types or manually update `database.ts`.

**Error handling UX:** Replace `alert()` and `confirm()` with a toast notification library.

**React anti-pattern:** `CreatePOFromBOMModal.tsx` calls `setSupplierId()` inside `useMemo`, which is a side effect in a pure computation. Move to `useEffect`.

**Error boundaries:** No React error boundaries exist. A crash in one component takes down the entire app.

**Currency inconsistency:** Default is USD in some places and INR in others (PartFormModal defaults to INR, ProjectAddPartModal defaults to USD, SQL schemas default to USD). Pick one and use consistently, or make it a config constant.

**Form validation:** Zod is installed but unused. Add validation schemas for part forms, supplier forms, and project forms.

**File upload:** The FileUpload component exists but is not wired into PartFormModal. The form shows a placeholder "Upload CAD/PDF files" area but clicking it does nothing.

## 10. Low: Performance and Scale

**Pagination:** Every table fetches ALL rows. Add cursor-based pagination with a configurable page size.

**Debounced search:** Search filters trigger on every keystroke. Add 300ms debounce.

**Query key specificity:** `queryClient.invalidateQueries({ queryKey: ['parts'] })` invalidates all part queries across all categories unnecessarily.

### Script 3: Rebuild project_parts

```sql
DROP TABLE IF EXISTS project_parts CASCADE;

CREATE TABLE project_parts (
  id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  project_section_id BIGINT NOT NULL REFERENCES project_sections(id) ON DELETE CASCADE,
  mechanical_manufacture_id BIGINT REFERENCES mechanical_manufacture(id) ON DELETE SET NULL,
  mechanical_bought_out_part_id BIGINT REFERENCES mechanical_bought_out(id) ON DELETE SET NULL,
  electrical_manufacture_id BIGINT REFERENCES electrical_manufacture(id) ON DELETE SET NULL,
  electrical_bought_out_part_id BIGINT REFERENCES electrical_bought_out(id) ON DELETE SET NULL,
  pneumatic_bought_out_part_id BIGINT REFERENCES pneumatic_bought_out(id) ON DELETE SET NULL,
  quantity INT NOT NULL DEFAULT 0,
  unit_price NUMERIC(18, 2) NOT NULL DEFAULT 0,
  currency TEXT DEFAULT 'INR',
  discount_percent NUMERIC(5, 2) NOT NULL DEFAULT 0,
  base_price_at_assignment NUMERIC(18, 2),
  supplier_name_at_assignment TEXT,
  reference_designator TEXT,
  notes TEXT,
  use_date_time TIMESTAMPTZ DEFAULT NOW(),
  assigned_at TIMESTAMPTZ DEFAULT NOW(),
  created_date TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_date TIMESTAMPTZ,
  CONSTRAINT exactly_one_part_fk CHECK (
    (CASE WHEN mechanical_manufacture_id IS NOT NULL THEN 1 ELSE 0 END +
     CASE WHEN mechanical_bought_out_part_id IS NOT NULL THEN 1 ELSE 0 END +
     CASE WHEN electrical_manufacture_id IS NOT NULL THEN 1 ELSE 0 END +
     CASE WHEN electrical_bought_out_part_id IS NOT NULL THEN 1 ELSE 0 END +
     CASE WHEN pneumatic_bought_out_part_id IS NOT NULL THEN 1 ELSE 0 END) = 1
  )
);

CREATE INDEX idx_pp_section ON project_parts(project_section_id);
CREATE INDEX idx_pp_mm ON project_parts(mechanical_manufacture_id) WHERE mechanical_manufacture_id IS NOT NULL;
CREATE INDEX idx_pp_mbo ON project_parts(mechanical_bought_out_part_id) WHERE mechanical_bought_out_part_id IS NOT NULL;
CREATE INDEX idx_pp_em ON project_parts(electrical_manufacture_id) WHERE electrical_manufacture_id IS NOT NULL;
CREATE INDEX idx_pp_ebo ON project_parts(electrical_bought_out_part_id) WHERE electrical_bought_out_part_id IS NOT NULL;
CREATE INDEX idx_pp_pbo ON project_parts(pneumatic_bought_out_part_id) WHERE pneumatic_bought_out_part_id IS NOT NULL;

CREATE TRIGGER update_project_parts_updated_date
    BEFORE UPDATE ON project_parts
    FOR EACH ROW
    EXECUTE FUNCTION update_updated_date_column();

ALTER TABLE project_parts ENABLE ROW LEVEL SECURITY;
CREATE POLICY "Authenticated users full access" ON project_parts
  FOR ALL USING (auth.role() = 'authenticated')
  WITH CHECK (auth.role() = 'authenticated');
```

### Script 5: Dashboard stats RPC

```sql
CREATE OR REPLACE FUNCTION get_dashboard_stats()
RETURNS JSON
LANGUAGE plpgsql
SECURITY DEFINER
AS $$
DECLARE
  result JSON;
  mm_count INT; mbo_count INT; em_count INT; ebo_count INT; pbo_count INT;
  low_stock INT; total_projects INT; active_projects INT;
  completed_projects INT; on_hold_projects INT;
  total_suppliers INT; pending_pos INT; total_pos INT;
BEGIN
  SELECT COUNT(*) INTO mm_count FROM mechanical_manufacture;
  SELECT COUNT(*) INTO mbo_count FROM mechanical_bought_out;
  SELECT COUNT(*) INTO em_count FROM electrical_manufacture;
  SELECT COUNT(*) INTO ebo_count FROM electrical_bought_out;
  SELECT COUNT(*) INTO pbo_count FROM pneumatic_bought_out;
  SELECT COUNT(*) INTO low_stock FROM (
    SELECT id FROM mechanical_manufacture WHERE min_stock_level > 0 AND stock_quantity <= min_stock_level
    UNION ALL SELECT id FROM mechanical_bought_out WHERE min_stock_level > 0 AND stock_quantity <= min_stock_level
    UNION ALL SELECT id FROM electrical_manufacture WHERE min_stock_level > 0 AND stock_quantity <= min_stock_level
    UNION ALL SELECT id FROM electrical_bought_out WHERE min_stock_level > 0 AND stock_quantity <= min_stock_level
    UNION ALL SELECT id FROM pneumatic_bought_out WHERE min_stock_level > 0 AND stock_quantity <= min_stock_level
  ) sub;
  SELECT COUNT(*) INTO total_projects FROM projects;
  SELECT COUNT(*) INTO active_projects FROM projects WHERE status IN ('planning','design','build','testing');
  SELECT COUNT(*) INTO completed_projects FROM projects WHERE status = 'completed';
  SELECT COUNT(*) INTO on_hold_projects FROM projects WHERE status = 'on_hold';
  SELECT COUNT(*) INTO total_suppliers FROM suppliers;
  SELECT COUNT(*) INTO pending_pos FROM purchase_orders WHERE status = 'Pending';
  SELECT COUNT(*) INTO total_pos FROM purchase_orders;
  result := json_build_object(
    'total_parts', mm_count+mbo_count+em_count+ebo_count+pbo_count,
    'mechanical_manufacture', mm_count, 'mechanical_bought_out', mbo_count,
    'electrical_manufacture', em_count, 'electrical_bought_out', ebo_count,
    'pneumatic_bought_out', pbo_count, 'low_stock_alerts', low_stock,
    'total_projects', total_projects, 'active_projects', active_projects,
    'completed_projects', completed_projects, 'on_hold_projects', on_hold_projects,
    'total_suppliers', total_suppliers, 'pending_pos', pending_pos, 'total_pos', total_pos
  );
  RETURN result;
END;
$$;
GRANT EXECUTE ON FUNCTION get_dashboard_stats() TO authenticated;
```
